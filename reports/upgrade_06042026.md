# MidcurveNN — Codebase Upgrade Report
**Date:** 06-Apr-2026  
**Scope:** `src/` folder only (excludes `publications/`, `references/`)

---

## 📋 Project Understanding

MidcurveNN is an academic research codebase implementing three approaches to compute the midcurve (medial axis) of 2D polygons using deep learning. The primary audience is researchers in geometric deep learning and CAD. Phase I (image-based) is largely functional; Phase II (GNN) is an incomplete stub; Phase III (LLM) was moved out of this repo.

The `src/` folder is organised as a collection of independent experiment modules, each following a `build_*_model.py` / `main_*.py` pattern. A shared `utils/` folder is intended for common data loading and plotting code, but in practice the utilities are partially duplicated. Configuration is split across `src/config.py` and `src/unet/config.py`.

## 🚨 Critical Issues

### 1. `NameError` when loading saved models (3 files)
**Files:** `simpleencoderdecoder/build_simple_encoderdecoder_model.py:102`, `denseencoderdecoder/build_dense_encoderdecoder_model.py:87`, `cnnencoderdecoder/build_cnn_encoderdecoder_model.py:138`

In all three, `print_best_metrics(metrics_history.history, ...)` is called unconditionally after the `if retrain_model / else load_model` block, but `metrics_history` is only instantiated inside the `if` branch. When a saved model is loaded from disk, this raises `NameError: name 'metrics_history' is not defined`.

```python
# current (broken on load path)
if not ... or retrain_model:
    metrics_history = MetricsHistory()
    ...
else:
    self.autoencoder = load_model(...)

print_best_metrics(metrics_history.history, ...)  # ← NameError if else branch taken
```

**Fix:** Move `print_best_metrics` inside the `if` block, or initialise `metrics_history = None` before the conditional and guard the call.

### 2. `ImportError` in `main_cnn_encoderdecoder.py`
**File:** `cnnencoderdecoder/main_cnn_encoderdecoder.py:10`

from utils.utils import plot_results  # ← utils.utils has no plot_results

`utils.utils` only contains `get_training_data` and `read_input_image_pairs`. `plot_results` lives in `utils.prepare_plots`. Running this script will immediately raise `ImportError`.

### 5. `denoiser_encoderdecoder.predict()` runs the model twice redundantly
**File:** `denoiserencoderdecoder/build_denoiser_encoderdecoder_model.py:93–97`

```python
def predict(self, test_noisy_images):
    png_profile_images = self.process_images(test_noisy_images)
    encoded_imgs = self.denoiser_autoencoder.predict(png_profile_images)   # run 1: noisy → denoised
    decoded_imgs = self.denoiser_autoencoder.predict(encoded_imgs)          # run 2: denoised → ??? (wrong)
    return test_noisy_images, decoded_imgs
```

The second call feeds denoised output back through the denoiser. The intended behaviour is to return the denoised result from run 1. Variable name `encoded_imgs` is also misleading (this is a full autoencoder, not just an encoder).

**Fix:** Remove the second predict call; return `encoded_imgs` directly.

### 6. `generate_sequences()` blocks on `plt.show()` per shape
**File:** `utils/prepare_data.py:416`

```python
for profile_dict in profiles_dict_list:
    ...
    plot_profile_dict(profile_dict)   # ← calls plt.show(), blocking interactive pop-up
```

`generate_sequences()` is designed to be called programmatically (e.g., `recreate_data=True`), but it calls `plot_profile_dict()` which contains `plt.show()`, blocking execution until each plot window is manually closed. This is clearly unintentional in an automated data-generation function.

**Fix:** Remove the `plot_profile_dict(profile_dict)` call from `generate_sequences()`.

## ♻️ Redundancy Findings

### Duplicate utility functions across two files

| Function | `utils/prepare_data.py` | `utils/utils.py` |
|----------|------------------------|-----------------|
| `read_input_image_pairs()` | ✅ canonical | Duplicate (identical body) |
| `get_training_data()` | ✅ canonical | Duplicate (identical body) |
| `plot_results()` | — | Configurable-size version |

`utils/utils.py` also duplicates `plot_results` from `utils/prepare_plots.py`, except `prepare_plots.plot_results()` hardcodes `100×100` while `utils.utils.plot_results()` accepts a `size` parameter — the one in `utils.py` is the better version. 

**Recommendation:** Delete `utils/utils.py`. Move the `size`-parameterised `plot_results` into `utils/prepare_plots.py` to replace the hardcoded version. Update all importers to use `utils.prepare_data` and `utils.prepare_plots`.

### Duplicated config files

`src/config.py` and `src/unet/config.py` are identical in every variable (same `TWO_STAGE`, `IMG_SHAPE`, `TRAIN_SIZE`, etc.). The UNet modules already resolve this correctly when run from `src/unet/` (they import `config` from the local directory). However, `src/config.py` also defines folder paths (`BASE_DIR`, `RAW_DATA_FOLDER`, etc.) that `src/unet/config.py` does not — meaning they serve different scopes. At minimum, the UNet-specific config should import from `src/config.py` instead of repeating hyperparameters.

### Commented-out dead code blocks

- `utils/prepare_data.py`: Large block of commented-out code (lines 38–49) after `combine_images()` — an earlier PIL-based implementation superseded by the numpy approach above it.
- `unet/test.py`: Commented `BASE_DIR` line (line 16) left from refactoring.
- `img2img/main_img2img_pytorch.py`: Several unused loss-list variables initialised but never appended to correctly (`Gen_losses = Gen_GAN_losses = Gen_L1_losses = []` all point to the same list object on line 234 — likely a bug too).

## 💡 Improvement Suggestions

### 1. `prepare_plots.py`: `plot_results()` hardcodes 100×100
**File:** `utils/prepare_plots.py:18,24`

```python
plt.imshow(original_imgs[i].reshape(100, 100), ...)
```

This crashes for 128×128 or 256×256 images. The `utils/utils.py` version accepts a `size` parameter — that approach should be adopted here.

### 2. `matplotlib.use('TKAgg')` should not be hardcoded at import time
**Files:** `utils/prepare_plots.py:8`, `utils/utils.py:17`

`matplotlib.use('TKAgg')` must be called before any other matplotlib imports and will fail on headless environments (training servers, CI). This should either be removed (let the caller control the backend) or made conditional:

```python
import os
if os.environ.get('DISPLAY'):  # only on systems with a display
    import matplotlib
    matplotlib.use('TKAgg')
```

### 3. `unet/train.py`: Loaded model is never recompiled before training
**File:** `unet/train.py:48–53`

When `load=True`, `gen_model` is reconstructed from JSON and weights are loaded, but `compile()` is never called. The model would be in an uncompiled state when `train_on_batch()` is called. Add compilation after weight loading (same optimiser and loss as in `init()`).

### 4. `img2img/main_img2img_pytorch.py`: Loss lists aliased to the same object
**File:** `img2img/main_img2img_pytorch.py:234`

```python
Disc_losses = Gen_losses = Gen_GAN_losses = Gen_L1_losses = []
```

This assigns all four variables to the same list object. Appending to any one of them appends to all. Use separate list literals: `Disc_losses = []; Gen_losses = []; ...`

### 5. `pix2pix/keras_gan_pix2pix.py`: Pix2Pix U-Net output uses `tanh` but loss expects `[0,1]`
**File:** `pix2pix/keras_gan_pix2pix.py:124`

The generator's final layer uses `tanh` (output range `[-1, 1]`), but the combined model is compiled with `mae` loss against images that were loaded and rescaled to `[0, 1]` by the data loader. The range mismatch means training converges incorrectly. The generator output should use `sigmoid`, or images should be normalised to `[-1, 1]`.

### 6. Consider adding `__init__.py` to model subdirectories
Currently only `src/utils/` and `src/unet/` have `__init__.py`. The other experiment folders (`simpleencoderdecoder/`, `cnnencoderdecoder/`, etc.) lack them. While the `sys.path.append(project_root)` workaround in `main_*.py` files works, adding `__init__.py` would make the package structure explicit and importable without path manipulation.

## ✅ What's Already Good

- **Consistent module pattern**: Every approach follows `build_*_model.py` + `main_*.py`, making it easy to add new experiments.
- **`MetricsHistory` callback** (`utils/metric_utils.py`): Well-designed Keras callback that captures best-epoch, training time, and comparison plots. Clean separation from model code.
- **`prepare_data.py` augmentation pipeline**: The rotation+translation+mirroring generation in `generate_images()` is clean and reusable. The use of DrawSVG for rasterisation keeps vector fidelity before pixel conversion.
- **UNet architecture** (`unet/unet.py`): The two-stage design with an auxiliary decoder and CoordConv inputs is architecturally sound and well-motivated. The `weighted_cross_entropy` implementation in `unet/utils.py` correctly handles both balanced and unbalanced variants with a clear docstring.
- **`src/config.py` path handling**: Uses `os.path` correctly to build all folder paths relative to `__file__` — no hardcoded paths (outside `create_ds.py`).
- **`prepare_plots.py` `plot_list_of_lines()`**: Flexible multi-subplot line visualiser with proper input validation and colour broadcasting — evidently added to support LLM output comparison and is well-written.
- **Sequence transformations** (`prepare_data.py`): `scale_sequence`, `rotate_sequence`, `translate_sequence`, `mirror_sequence` all use proper 2D homogeneous transformation matrices — mathematically correct.
