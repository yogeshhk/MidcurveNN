\begin{frame}[fragile]\frametitle{Midcurve : The Problem}

	\begin{itemize}
	\item {\bf Goal}: Given a 2D closed shape (closed polygon) find its midcurve (polyline, closed or open)
	\item {\bf Input}: set of points or set of connected lines, non-intersecting, simple, convex, closed polygon
	\item {\bf Output}: another set of points or set of connected lines, open/branched polygons possible
	\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Midcurve : Graph 2 Graph (The Ideal)}

  \begin{columns}[t]
    \begin{column}[T]{0.55\linewidth}
      \begin{itemize}
        \item {\bf Ideal framing}: Input profile graph $\rightarrow$ Output midcurve graph
        \item Both graphs have \textbf{different topologies}: different node/edge counts, positions
        \item Standard node-prediction and link-prediction GNNs assume similar input/output topology; not suitable here
        \item Direct graph-to-graph translation requires a coarsening / dimension-reduction formulation
        \item \alert{Phase III} of this work tackles this directly using a Graph Transformer
      \end{itemize}
    \end{column}
    \begin{column}[T]{0.45\linewidth}
      \centering
      \includegraphics[width=\linewidth,keepaspectratio]{midcurve_as_graph}
    \end{column}
  \end{columns}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Midcurve $==$ Dimension Reduction / Translation}

	\begin{itemize}
	\item Like PCA, find the principal curve that \textit{represents} the profile shape
	\item Left side (input): 2D Sketch Profile; Right side (output): 1D Midcurve
	\item \textbf{Not} an Auto-Encoder: input and output are \textit{different} shapes
	\item Variable-length input/output makes padding unsafe in geometry (any value is valid)
	\item Analogous to \textbf{graph summarization}: reduce a large graph to a smaller graph preserving structural essence, like text summarization keeps the meaning
	\end{itemize}
\begin{center}
\includegraphics[width=0.55\linewidth,keepaspectratio]{midcurve20}
\end{center}
\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Why Padding Fails in Geometry}

	\begin{itemize}
	\item OK for NLP, say Machine Translations, where padding values like ``-1'' can be added along with other words (vectors or indices)
	\item But in Geometry, its not OK.
	\item Because any value can represent a Valid Input, even though we don't want it to be the input.
	\item Not all shapes can be serialised as sequences: Y-shapes and concentric rings (\textit{O}) cannot be traversed without lifting the pen, ruling out Seq2Seq models entirely
	\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Three Phase Strategy}

This trade-off motivates the \textbf{three-phase} strategy:
		\begin{itemize}
		\item \textbf{Phase I}: Raster Image $\rightarrow$ Image (encoder-decoder variants)
		\item \textbf{Phase II}: BRep JSON via LLMs, exact coords, current best performer
		\item \textbf{Phase III}: Geometry graph $\rightarrow$ graph, exact, native branches
		\end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Phase II: Text as a Medium for Geometry}

  \begin{itemize}
    \item Phase I approximates geometry as pixel images: \alert{raster dilution}
    \item Phase II takes a different route: serialize geometry as \textbf{structured JSON text}
    \item LLMs can read and write JSON naturally: frame midcurve as \textbf{machine translation}:
      \begin{itemize}
        \item {\small Input: Profile geometry $\rightarrow$ BRep JSON text}
        \item {\small Output: Midcurve geometry $\leftarrow$ predicted BRep JSON text}
      \end{itemize}
    \item \textbf{Advantage over Phase I}: output is \textit{exact point coordinates}, not noisy pixels: no raster post-processing needed
    \item \textbf{Trade-off}: LLMs are probabilistic; output may occasionally need syntactic repair
    \item Models explored: Qwen2.5-7B, Gemma, Mistral (QLoRA fine-tuning); Nemotron-Mini-4B; few-shot prompting
  \end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{BRep: Boundary Representation in JSON}

  \textbf{Geometry encoded as a 3-level hierarchy} (2D adaptation of solid-modelling B-rep):

  \begin{columns}[t]
    \begin{column}[T]{0.48\linewidth}
      \begin{itemize}
        \item \textbf{Points}: ordered list of $(x,y)$ coordinates
        \item \textbf{Lines}: pairs of point indices: encodes \textit{connectivity}
        \item \textbf{Segments}: chains of line indices: encodes \textit{continuous open curves}; \textbf{multiple Segments} represent junctions in T- and Plus-shapes
      \end{itemize}
      \vspace{0.5em}
      {\small Key invariant: only \textit{coordinates} change under geometric transforms; topology (Lines, Segments) stays fixed}
    \end{column}
    \begin{column}[T]{0.52\linewidth}
      {\footnotesize\textbf{I-shape Profile:}}\\[0.15em]
      {\footnotesize
        \texttt{Points:} $(5,5),(10,5),(10,20),(5,20)$\\
        \texttt{Lines:}  $[0,1],[1,2],[2,3],[3,0]$\\
        \texttt{Segs:}   $[0,1,2,3]$
      }\\[0.6em]
      {\footnotesize\textbf{I-shape Midcurve:}}\\[0.15em]
      {\footnotesize
        \texttt{Points:} $(7.5,5),(7.5,20)$\\
        \texttt{Lines:}  $[0,1]$\\
        \texttt{Segs:}   $[0]$
      }\\[0.6em]
      {\footnotesize\textbf{T-shape Midcurve (branched):}}\\[0.15em]
      {\footnotesize
        \texttt{Segs:} $[0],[1],[2]$ \quad (3 arms = 3 Segments)\\
        \textit{Sequences cannot encode this; BRep can.}
      }
    \end{column}
  \end{columns}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Phase II: Results}

  \begin{itemize}
    \item \textbf{Qwen2.5-7B (QLoRA fine-tuning)}: \alert{current best performer}:
      \begin{itemize}
        \item MAE = 0.78, RMSE = 1.24, Hausdorff distance = 2.1
        \item \textbf{PSR (Profile Shape Reconstruction) = 98\%} on 100-sample test set
      \end{itemize}
    \item \textbf{Nemotron-Mini-4B (2-shot, no fine-tuning)}:
      \begin{itemize}
        \item PSR = 85.7\%, topology accuracy = 0.83
        \item Fits in $\sim$2\,GB VRAM: deployable on consumer hardware
      \end{itemize}
    \item \textbf{Key advantage}: output is exact \textit{point coordinates}: no pixel clean-up
    \item \textbf{Limitation}: branched shapes (T, Plus) require well-formed multi-Segment JSON output
  \end{itemize}

\end{frame}

\begin{frame}[fragile]\frametitle{Phase III: Back to Pure Geometry: Graph to Graph}

  \begin{columns}[t]
    \begin{column}[T]{0.55\linewidth}
      \begin{itemize}
        \item Recall the ideal: input profile graph $\rightarrow$ output midcurve graph
        \item Phase III implements this directly: \textbf{no pixels, no text tokens}
        \item \textbf{Input}: polygon as graph: nodes = corner points with $(x,y)$; edges = adjacent corners
        \item \textbf{Output}: midcurve as graph: different node/edge counts and positions
        \item \textbf{Advantage}: exact coordinates, native branch handling, zero post-processing
        \item Branches (T, Plus) are naturally captured in the graph topology
      \end{itemize}
    \end{column}
    \begin{column}[T]{0.45\linewidth}
      \centering

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Phase III: Graph Transformer Architecture}

  \begin{itemize}
    \item \textbf{Core challenge}: graph-to-graph translation with \textit{different} input/output topology: not standard node/link prediction
    \item \textbf{Non-auto-regressive Graph Transformer} (trained from scratch):
      \begin{itemize}
        \item LaplacianPE positional encoding $\rightarrow$ TransformerConv layers $\rightarrow$ TopKPooling
        \item TopKPooling acts as graph \textit{coarsening}: the geometric dimension-reduction step
        \item Loss: Chamfer distance (point-cloud matching) + BCE (adjacency matrix)
      \end{itemize}
    \item \textbf{Second variant}: pretrained Graphormer (HuggingFace) fine-tuned in two phases (frozen head $\rightarrow$ joint fine-tuning)
    \item Operates on raw $(x,y)$ coordinates: no rasterization, no JSON encoding
    \item Status: \alert{implemented, evaluation ongoing}
  \end{itemize}

\end{frame}

%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%%
\begin{frame}[fragile]\frametitle{Conclusion}

  \begin{itemize}
    \item Midcurve is a \textbf{geometric dimension-reduction} problem: 2D profile $\rightarrow$ 1D skeleton (graph $\rightarrow$ smaller graph)
    \item \textbf{Phase I} (image-based): encoder-decoder on raster images --- feasible; CNN encoder-decoder is the best primary variant; rasterization trade-off limits precision
    \item \textbf{Phase II} (LLM / text-based): BRep JSON + LLM fine-tuning: \alert{current best performer}: PSR = 98\%, exact coordinates, branching via Segments
    \item \textbf{Phase III} (geometry-based): Graph Transformer on polygon graphs: the ideal formulation; no approximation, native branch handling; evaluation ongoing
    \item \textbf{Next frontier}: generalise to arbitrary 2D thin polygons; extend the graph-to-graph paradigm toward 3D midsurface computation
  \end{itemize}

\begin{center}
\alert{\large Graph-to-Graph is the ultimate goal for MidcurveNN.}
\end{center}

\end{frame}
